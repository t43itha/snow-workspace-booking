It's a command-line toolkit (the package is @​servicenow/sdk, and you typically run it with now-sdk) that lets you create, build, and deploy scoped applications from your local machine. You write code in Fluent, which is ServiceNow's own code framework for defining platform metadata. The output is native code: tables, business rules, script includes, flows. All of it compiles down to the same scoped app you'd get if you built it in ServiceNow Studio. Why Fluent matters: because you're writing in a language the platform already understands, there's no translation layer that breaks things. The code you write maps directly to the metadata ServiceNow expects. If you've ever had a deployment go sideways because some import didn't map correctly, you probably appreciate that directness. What is git Git shows up in just about every modern coding workflow. If you've never actually used it, the terminology can feel like everyone else got a memo you missed. Here's the two-minute crash course on git. Git is a snapshot tool. It saves your project at specific moments in time and lets you move between those moments. Every snapshot is called a commit. Commits stack up permanently. Nothing gets overwritten. COMMITS = SNAPSHOTS ┌───────────────────────────────────┐ │ "Added assignment rule" │ ← latest ├───────────────────────────────────┤ │ "Added feedback table" │ ├───────────────────────────────────┤ │ "Initial project setup" │ ← oldest └───────────────────────────────────┘ Permanent. Nothing gets overwritten. Three zones to understand. The working tree is the files on your machine that you're actively editing. The staging area is a holding space for changes you want to include in your next snapshot. Committed history is the permanent record of every snapshot you've saved. Changes move through those zones in order: you edit in the working tree, stage the changes you care about, then commit to history. WHERE YOUR CODE LIVES ┌──────────────┐ │ Working tree │ ← you edit here └──────┬───────┘ │ stage changes ┌──────v───────┐ │ Staging area │ ← changes queued └──────┬───────┘ │ commit ┌──────v───────┐ │ History │ ← permanent record └──────────────┘ All of that lives on your machine. GitHub or Gitlab or BitBucket is separate. There are remote hosts. It stores a copy of your commit history in the cloud, which gives you a backup and the ability to sync code across machines. When you push code to GitHub, you're sending your local history to that remote copy. TWO DIFFERENT THINGS ┌──────────────────┐ push ┌──────────────┐ │ Your machine ├─────────────> │ GitHub │ │ │ │ │ │ git │ <──────────── │ remote copy │ │ (local history) │ pull │ (backup) │ └──────────────────┘ └──────────────┘ git = version control tool GitHub = remote host for your history So keep this mental model: git tracks your project history locally. GitHub stores and shares that history remotely. All of this matter because your commit history is your audit trail. If something breaks, you can look back and see exactly what changed and when. If you introduce a bug, you can roll back to a working state. The further you get into a project without that safety net, the more it hurts when something goes sideways.
